# Design Principles — 구현/리뷰 코딩 표준

P5(구현)가 따르고 P8(리뷰)가 검사하는 코드 표준. RTM의 REQ를 코드로 옮길 때의 규율.
CLAUDE.md의 Core Principles를 코드 레벨로 구체화한 것이다.

## 1. TDD 규율
- **RED 먼저**: 실패하는 테스트(P4) 없이 구현(P5)을 작성하지 않는다.
- 구현은 **테스트를 통과시키는 최소한**으로. 요구되지 않은 기능을 추측해 만들지 않는다.
- 모든 코드 변경은 RTM의 **REQ-ID로 추적**된다 — 매핑되지 않는 코드는 작성 금지.

## 2. File = Interface
- Phase 간 상태는 **파일로만** 전달한다. 인메모리/암묵적 컨텍스트에 의존하지 않는다.
- 출력 파일은 다른 Phase·에이전트가 다시 읽을 것을 전제로 **스스로 완결적**이어야 한다.

## 3. 검증의 진실성 (Constraint Verification / Real E2E)
- 외부 의존성(API·DB·파일시스템 등)은 **실제 호출로 검증**한다.
- E2E(P6)에서 **mock 금지**. mock은 unit 레벨에서만, 경계가 명확할 때.
- "통과한 것처럼" 보이게 만들지 않는다. 실패는 실패로 드러낸다 (→ hooks가 harness에서 강제).

## 4. 코드 품질
- 명확한 네이밍 > 주석. 의도가 코드로 드러나게 한다.
- 예외/에러 경로를 무시하지 않는다. 삼키는 `catch`/빈 `except` 금지.
- 결정적(deterministic) 동작 우선 — 시간·랜덤·순서 의존은 테스트 가능하게 주입.
- DRY: 동일 로직 3회 반복 전에 추출. 단, 성급한 추상화도 경계.

## 5. 주변 코드와의 일관성
- 새 코드는 **주변 코드처럼** 보이게 한다 — 네이밍·구조·에러 처리·주석 밀도를 맞춘다.
- 기존 컨벤션이 표준과 충돌하면, 임의로 바꾸지 말고 RTM/리뷰에 근거를 남긴다.
